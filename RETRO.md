# Retrospective

## What Went Well

- Good collaboration as a team
    - We all contributed equally with PRs which helped us keep the quality of the handbook good and not let through any typos etc.
- Effective use of Git
    - We followed the assignment brief of using a trunk-based development approach using branches and pull requests. This is something some members on our team hadn't used before so it was useful for our team members to help teach eachother. Even those who had used git before it helped spot gaps in their knowledge. This helped ensure that all changes were reviewed before merging.
- Made a clear structure and plan before starting anything
    - We created Git issues so everyone knew exactly which tasks we were all working on and when. This made it easier for everyone to contribute without confusion.

## What we learned

- Improved Git skills
    - Even though some of us had used Git during internships, we all still learned new things in this assignment.
- Importance of small pull requests
    - Allowed for faster reviews as it didn't take as long for other members to review each others work. This kept progress consistent across all our work.
- How important planning is
    - Having a project plan at the start helped us stay organised and ensured that tasks were distributed fairly.

## What Didn't Go Well

- Delays in PRs
    - Sometimes team members were waiting for reviews before they could carry on with other work. This showed us scenarios devs have to go through when being blocked. This sometimes caused small delays but through team work we got it all done
- Inconsistent formatting
    - Often we didn't make it clear with our structure. At the start in planning we created a template for each of us to fill in but yet we came out with slightly different ways in which we formatted things but these were resolved in our code reviews.
- Too much focus on individual sections early on
    - We worked quite independently at the start, which meant the document didn’t feel fully connected until later.
- Difficulty identifying common themes across sources
    - It was sometimes hard to not just summarise our sources instead of just getting the shared best practices.
- Not all PRs done fully
    - Some reviews were often brief by certain members or just accepted.
- Small inconsistencies in writing style
    - Sometimes we had different tones and even at the end we had an issue with if we wanted to speak in the first person or not.

## What We Would Do Differently Next Time

- Set a deadline for PRs
    - Instead of having to wait hours for PRs try get to them as soon as possible.
- Make a clearer formatting structure at the start
    - We created a base template to fill in our research but still ended up formatting it slightly different. Next time we would make a more rigid template for us to fill in so there are no descrepancies.
- Naming conventions
    - We agreed upon a plan at the start but we never worked in strictly many naming conventions for things like branch naming or commit messages. If we made these clear at the start it could've made things slightly easier as it looks cleaner.
- Use draft pull requests
    - If we created draft PRs this would allow us to get earlier feedback before our work is fully finished which reduces the number of changes we need to make later

## Overall

We all agreed in saying that this assignment was hugely valuable to us in understanding how a team should work and use Git together in a real development environment. Even though some of us have used Git on our internships we still found this assignment valuable
as teaching other team members helped us spot gaps in our own knowledge. We all feel that this kind of assignment would've been even better for us if we were given something similar in first year as this is the first time some of us or our classmates have touched Git. Not only has this improved our understanding of Git in a team workflow, we improved our knowledge
of how to do a good code review, automation testing and task estimation.
